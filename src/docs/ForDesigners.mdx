## What is SKAI UI?

SKAI UI is our **single source of truth** for all user interface components, design tokens, patterns, and guidelines. Everything in this Storybook directly corresponds to our production code.

### Key Principles

| Principle         | Description                                                       |
| ----------------- | ----------------------------------------------------------------- |
| **Consistency**   | Same components used everywhere, same behavior expected           |
| **Accessibility** | WCAG 2.1 AA compliant, keyboard navigable, screen reader friendly |
| **Performance**   | Optimized bundle size, lazy loading, minimal re-renders           |
| **Flexibility**   | Components adapt to different contexts via variants and slots     |

### üîÑ Figma Token Sync Workflow

We use **Tokens Studio for Figma** to manage design tokens. Here's the complete sync process:

#### Setup (One-time)

1. Install [Tokens Studio](https://tokens.studio/) plugin in Figma
2. Create a new token set named "SKAI"
3. Import base tokens from `design-tokens.json` in the codebase

#### Making Token Changes

1. **In Figma:** Update tokens using Tokens Studio plugin
2. **Export:** Click "Export" ‚Üí Save as JSON
3. **Save file:** Copy to `modules/skai-ui/design-tokens.figma.json`
4. **Sync:** Run the sync command:

```bash
cd modules/skai-ui
npm run tokens:sync
```

5. **Verify:** Check generated files:
   - `src/lib/design-tokens.json` - Token definitions
   - `src/lib/design-tokens.css` - CSS custom properties

#### Validation

Before deploying, validate tokens match Figma:

```bash
npm run tokens:validate
```

This will show any discrepancies between Figma export and code.

#### Token File Structure


### Component Workflow

When designing a new component:

1. **Design in Figma** - Create component with all states
2. **Document specs** - Note sizes, spacing, behavior
3. **Review with dev** - Align on props and variants
4. **Dev implements** - Component added to skai-ui
5. **Design QA** - Verify implementation matches design
6. **Ship!** - Available in Storybook and production

## Component Anatomy

Every component should have documented:

### States

- **Default** - Normal resting state
- **Hover** - Mouse over (desktop)
- **Active/Pressed** - While clicking/tapping
- **Focus** - Keyboard focused (visible ring)
- **Disabled** - Non-interactive
- **Loading** - Async operation in progress
- **Error** - Validation failed

### Variants

Example Button variants:

- `default` - Standard button
- `destructive` - Dangerous action
- `outline` - Secondary importance
- `ghost` - Tertiary, inline actions
- `link` - Text-only, navigational

### Sizes

Standard sizes across components:

- `sm` - Compact UI, tables, dense layouts
- `md` - Default, most contexts
- `lg` - Hero sections, primary actions

## Accessibility Requirements

All components must meet these requirements:

### Visual

- ‚úÖ 4.5:1 contrast ratio for text
- ‚úÖ 3:1 contrast ratio for UI elements
- ‚úÖ Never rely on color alone for meaning
- ‚úÖ Focus indicators visible

### Interactive

- ‚úÖ 44x44px minimum touch target
- ‚úÖ Keyboard navigable
- ‚úÖ Screen reader labels
- ‚úÖ Reduced motion support

### Testing

Designers should test with:

- Figma a11y plugins (Stark, A11y Annotation Kit)
- Color blindness simulators
- High contrast mode

## Handoff Checklist

When handing off designs to developers:

- [ ] All states designed (default, hover, active, disabled, error, loading)
- [ ] Responsive behavior specified (mobile, tablet, desktop)
- [ ] Spacing uses design tokens (not arbitrary values)
- [ ] Colors use semantic tokens (not hex values)
- [ ] Typography uses type scale
- [ ] Accessibility requirements noted
- [ ] Edge cases covered (empty states, errors, long text)
- [ ] Dark mode variant provided
- [ ] Animation/transition specs if applicable
